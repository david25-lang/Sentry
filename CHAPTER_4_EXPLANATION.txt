Chapter 4 Explanation: System Implementation and Integration

Project Overview
This work connected two separate parts of the project into one working system:
1. The FastAPI backend in the folder "shittu oluyinka"
2. The Next.js frontend dashboard in the folder "sentry"

The goal was to make both parts work together professionally so the frontend can send requests to the backend, receive model results, and display road damage analysis data in the UI.

What Was Done

1. Backend integration
The FastAPI backend was expanded so it now serves the routes the frontend expects. Before the changes, the backend only exposed a simple /predict endpoint and a basic health check. The frontend, however, was designed to call a richer API structure with endpoints such as:
- /health
- /api/v1/models
- /api/v1/classes
- /api/v1/detect/upload
- /api/v1/detect/url
- /api/v1/classify/upload
- /api/v1/classify/url
- /api/v1/compare/upload
- /api/v1/compare/url

The backend now returns structured JSON responses that match the frontend TypeScript types for detection, classification, and comparison.

2. Model connection
The backend was updated to use the trained road-damage model instead of the generic YOLO COCO weights. It now points to the trained weights file:
- runs/detect/runs/train/pothole-crack-normal-2/weights/best.pt

This means the system now returns the expected classes:
- pothole
- crack
- normal

3. Environment configuration
The backend environment file was updated so the model path is stored properly in:
- .env
- .env.example

This makes the setup easier to reproduce and keeps the configuration clear.

4. Frontend connection
The Next.js frontend was kept on the existing API base URL:
- http://localhost:8000

The frontend already had code for detection, classification, model comparison, and dashboard views, so once the backend routes were made compatible, the two projects started working together without a major rewrite.

5. Server startup and verification
Both servers were started and checked:
- Backend: http://localhost:8000
- Frontend: http://localhost:3000

The backend health endpoint was verified and confirmed that the trained model is loaded and the classes are correct.

Challenges Faced

1. API mismatch between frontend and backend
The frontend was built for a more complete API structure, while the backend initially only had a simple /predict endpoint. This caused a mismatch in request paths and response formats.

2. Wrong default model
The backend originally used yolov8n.pt, which is a generic COCO model. That model does not match the road-damage classes used in the frontend. The fix was to point the backend to the trained best.pt model.

3. Python version compatibility
The backend was running on Python 3.9. A modern type annotation style caused an import-time error, so the code had to be adjusted to work properly with Python 3.9.

4. Port conflicts
When starting the servers, some ports were already in use by earlier processes. The frontend was already running on port 3000, so it was verified instead of duplicated.

5. File and workspace separation
The project is split across two folders, so care was needed to make sure the backend and frontend were started from the correct directories.
